Bento SaaS（ポートフォリオプロジェクト）

概要

本プロジェクトは、お弁当サブスクリプション型SaaS を題材としたポートフォリオ用システムです。

限られた期間・リソースの中で、以下を示すことを目的としています。
	•	業務を中心に考えた Domain-Driven Design（DDD）思考
	•	責務に基づいた マイクロサービス分割
  •	テスト駆動開発(TDD)に沿った開発
	•	React を用いたフロントエンドと、複数言語バックエンドによる ポリグロット構成
	•	実装可能性を重視した 現実的なスコープ管理

完成度や見た目よりも、
「なぜこの設計にしたのかを説明できること」 を重視しています。

⸻

システム全体像

本システムは以下の構成を想定しています。
	•	フロントエンド：React
	•	バックエンド：複数の Bounded Context（BC）
	•	同期通信方式：HTTP（REST）
  •	非同期通信方式：Kafka

React UI はユーザー操作の窓口として機能し、
各 Bounded Context はそれぞれ独立した責務を持つサービスとして設計されています。

⸻

フロントエンド（React UI）

位置付け
	•	本番品質ではなく デモ可能な MVP UI
	•	バックエンド設計・業務フローを可視化するための UI

方針
	•	Hooks を用いたシンプルな実装
	•	状態管理や設計の複雑化は避ける
	•	「動くこと」「流れが分かること」を最優先

想定画面
	•	ログイン
	•	プラン一覧・詳細
	•	サブスクリプション状況確認
	•	請求書一覧
	•	通知一覧（インアプリ）

⸻

バックエンド構成（Bounded Context）

Plan / Subscription BC
	•	プラン作成・管理
	•	サブスクリプション申込・状態管理
	•	業務の中心となるドメイン

本プロジェクトでは、詳細実装よりも
責務分離と状態遷移の考え方 を重視しています。

⸻

Invoice BC（Python + AWS Lambda）
	•	請求書情報の管理
	•	ユーザーが請求状況を確認するための BC

決済処理や会計ロジックは扱わず、
UI に必要な最小限の請求情報提供 にスコープを限定しています。

⸻

Notification BC（Go）
	•	ユーザー向け通知の管理
	•	インアプリ通知を提供

通知は「送信」ではなく、
ユーザーに提示する情報を記録・提供するもの として扱っています。

メール通知やプッシュ通知は将来拡張とし、
本プロジェクトでは対象外としています。

⸻

通知の考え方

本システムにおける通知は、以下を意味します。
	•	プランの状態変化
	•	請求書発行などのシステムイベント
	•	ユーザーが UI 上で確認できるインアプリ通知

複雑な通知配信基盤は導入せず、
シンプルで説明しやすい構成 を採用しています。

⸻

スコープ管理について

本プロジェクトでは、以下を 意図的に実装対象外 としています。
	•	決済処理
	•	メール・プッシュ通知
	•	高度なエラーハンドリング
	•	パフォーマンス最適化

理由は、短期間での完成と
設計判断の明確化を優先するためです。

⸻

このプロジェクトで示したいこと
	•	技術選定を責務ベースで行っていること
	•	小さくても意味のあるマイクロサービス分割
	•	DDD に基づく Bounded Context 思考
  •	TDD により開発の進め方
	•	現実的なスコープコントロール能力

⸻
